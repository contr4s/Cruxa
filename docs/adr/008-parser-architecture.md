# ADR 008: Архитектура парсера данных (Phase 2)

## Статус
Active (Updated 2026-06-05)

## Дата
2026-06-04

## Контекст
Phase 1 завершена: есть полноценное REST API с базой данных PostgreSQL, но база скалодромов пуста — проблема холодного старта. Для наполнения БД реальными данными требуется инструмент сбора информации о скалодромах из внешних источников.

**Требования:**
- Собрать базу скалодромов Москвы и Санкт-Петербурга
- Данные должны включать: название, адрес, координаты, контакты, сайт, часы работы, цены, фото, площадь стен, макс. высоту, год основания, метро, теги
- Решение должно быть простым в запуске и не требовать постоянной инфраструктуры
- Спаршенные данные должны загружаться в БД через существующий API (чтобы пройти слой валидации и бизнес-логики)

## Рассмотренные варианты

### Вариант 1: Python-скрипт
- Быстро написать, но не вписывается в монорепозиторий .NET
- Требует отдельного рантайма на машине разработчика
- Сложнее переиспользовать существующие модели данных и утилиты

### Вариант 2: Worker Service внутри API
- Фоновый сервис, встроенный в `Cruxa.Api`
- Сложнее контролировать запуск (нужен триггер или API-эндпоинт)
- Потенциально влияет на стабильность основного приложения
- Нельзя запускать изолированно от API

### Вариант 3: CLI Console App — отдельный проект в солюшене
- Независим от API, можно запускать отдельно
- Простая архитектура: scrape → JSON → seed через API
- Данные проходят через существующую валидацию API
- Легко покрывается тестами

## Решение
**Выбран Вариант 3: CLI Console App (`Cruxa.Parser`).**

### Архитектура:
```
src/Cruxa.Parser/
├── Program.cs                    # Entry point, DI, command routing
├── Commands/
│   ├── ScrapeCommand.cs          # Сбор данных из climbingpro.ru
│   └── SeedCommand.cs            # Загрузка JSON через REST API
├── Models/
│   └── ParsedGym.cs              # DTO спаршенного скалодрома
├── Services/
│   ├── IClimbingProClient.cs     # Интерфейс клиента climbingpro.ru
│   ├── ClimbingProClient.cs      # HTML-парсер (HtmlAgilityPack) + координаты из data.json
│   └── GymExportService.cs       # Экспорт в JSON
└── appsettings.json
```

### Источник данных (Phase 2, Revised):
- **maps.climbingpro.ru** — каталог скалодромов России с детальными страницами
- **Сбор:** страница города → список ссылок на скалодромы → каждая страница парсится индивидуально
- **Парсинг:** HtmlAgilityPack для извлечения данных из HTML

### Детальные поля (Phase 1 expansion):
| Поле | Тип | Источник |
|------|-----|----------|
| Name | string | Заголовок страницы |
| Address | string | Лейбл "Адрес:" |
| Description | string | Первый абзац описания |
| ContactInfo | string | Ссылка `tel:` |
| ContactEmail | string? | Ссылка `mailto:` |
| Website | string? | Ссылка "ПЕРЕЙТИ НА САЙТ" |
| SocialLinks | List\<string\> | Ссылки VK/FB/IG/Youtube |
| WorkingHours | string? | Секция "Режим работы" |
| Prices | List\<PriceItem\> | Секция "Прайс-лист" |
| PhotoUrls | List\<string\> | Изображения `/media/photo/` |
| WallArea | double? | Regex "Площадь стен: X м²" |
| MaxHeight | double? | Regex "Макс. высота: X м" |
| YearFounded | int? | Regex "Год основания XXXX" |
| MetroStations | List\<string\> | Ссылки `/metro/` |
| Tags | List\<string\> | Лейбл "Теги:" |
| Latitude / Longitude | double? | `/data.json` с climbingpro.ru |

### Координаты:
- **Источник:** каждый скалодром на climbingpro.ru отдаёт файл `/data.json` в формате GeoJSON
- **URL:** `https://maps.climbingpro.ru/{gymPath}data.json`
- **Формат:** `{ "center": ["lat", "lon"], "features": [...] }`
- Извлекается поле `center` — оно гарантированно присутствует и совпадает с координатами метки на карте
- Координаты nullable — если data.json недоступен, остаются null (graceful degradation)

### Seeder:
- **API endpoint `POST /api/gyms/import`** на существующем API (Admin-only)
- Парсер собирает JSON → seed-команда читает JSON и отправляет POST через HTTP
- Данные проходят полную валидацию и бизнес-логику API
- Chunking по 50-100 записей для избежания таймаутов

## Обоснование
1. **Изоляция:** Парсер не влияет на стабильность API. Можно запускать на машине разработчика отдельно.
2. **Pipeline через API:** Данные проходят через существующую валидацию, дедупликацию и бизнес-логику — не нужно дублировать её в парсере.
3. **Тестируемость:** Чёткие границы между scrape → export → seed, каждый этап покрывается unit-тестами.
4. **Знакомый стек:** C#/.NET — разработчик не переключает контекст между языками.
5. **HtmlAgilityPack:** Стабильная, проверенная библиотека для HTML-парсинга в .NET.

## История источника данных

### Overpass API (OSM) — Phase 2, Initial
- **OpenStreetMap Overpass API** — поиск по тегу `sport=climbing`
- Покрытие: ~20-30 результатов на Москву
- **Недостатки:** нет цен, фото, площади стен, метро, года основания, соцсетей
- Решение: заменён на climbingpro.ru для более полных данных

### maps.climbingpro.ru — Phase 2, Current
- **URL:** `https://maps.climbingpro.ru/city/{slug}/` (например, `moskva`, `sankt-peterburg`)
- Парсинг статического HTML (Yandex Maps JS API не используется для данных)
- **Фильтрация:** исключаются ссылки футера (`/addsite/`, `/reclame/`, `/agreement/`, `/contact/`, `/rules/`) и ссылки на другие города
- **Ограничение:** координаты загружаются динамически через `/data.json` (не в HTML)
- **Лицензия:** данные сайта публично доступны, парсинг для некоммерческого использования

## Последствия
- `Cruxa.Parser` добавляется в `Cruxa.slnx` — теперь 5 проектов в солюшене
- При изменении API (например, эндпоинта импорта) нужно синхронизировать парсер
- Для работы парсера **не требуется** API-ключ
- Парсер не предназначен для production-деплоя — это dev-инструмент для наполнения БД
- Добавлена зависимость от `HtmlAgilityPack` (v1.12.0)
- Добавлена зависимость от `Cruxa.Domain` для `PriceItem`
- Координаты nullable — могут отсутствовать для некоторых заведений
- Требуется уважать rate limit при парсинге (1 сек между запросами)

## Что не входит
- Регулярный автоматический парсинг — только ручной запуск через CLI
- Другие источники — отложены до востребования
- Капча/антибот-защита — сайт не блокирует парсинг
