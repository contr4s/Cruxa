# Рабочий процесс и договоренности (Workflow & Agreements)

Этот документ фиксирует ключевые правила работы и стандарты качества, которым мы следуем при разработке Cruxa.

## 1. Строгое следование требованиям
- Любая написанная строчка кода, структура API или спроектированный интерфейс должны предельно точно соответствовать зафиксированной документации (`FunctionalRequirements.md`, `Design.md`, `Database.md`). 
- Никаких отклонений: если в процессе технической реализации возникает необходимость изменить логику (или появляется идея для улучшения), сначала мы обсуждаем это, **обновляем документацию**, и только затем пишется код.

## 2. Architecture Decision Records (ADR)
- Каждое значимое техническое решение (выбор конкретной библиотеки, архитектурного паттерна, способа хранения медиа, структуры проекта) обязательно документируется.
- Для этого будет использоваться папка `docs/adr/`, где каждый файл отвечает на вопросы: 
  - **Контекст** (какую проблему решаем)
  - **Рассмотренные варианты**
  - **Принятое решение**
  - **Обоснование** (почему именно так).

## 3. Тестирование Backend-а (Coverage)
- **Backend (C# ASP.NET Core):** Unit-тесты для доменной логики (GradeMapping, Guard Clauses). Integration-тесты через Testcontainers с реальным PostgreSQL.
- **Парсер:** Верифицирован вручную на 50+ реальных залах Москвы и СПб.
- **Frontend:** MSW для моков API на этапе разработки. Smoke-тесты ключевых пользовательских сценариев через браузер.

## 4. Итеративность и Чистота кода
- Двигаемся строго по шагам, описанным в `Roadmap.md`. Не прыгаем на разработку мобилки, пока не стабилизируем API.
- Код должен быть читаемым и самодокументируемым. Используем линтеры экосистем (EditorConfig / StyleCop для C#, ESLint для React, стандартный Dart Linter для Flutter).
- Сложные участки (особенно маппинги и алгоритмы) дополнительно снабжаются XMLDoc / JSDoc комментариями.
